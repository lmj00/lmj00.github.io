### 개요
> WebSocket 프로토콜은 HTTP Upgrade 기반의 핸드셰이크 뒤에 기본 프레이밍을 얹어, 단일 TCP 연결로 양방향 통신을 제공한다.

## 핸드셰이크

WebSocket 연결은 **클라이언트 → 서버** 순서의 HTTP Upgrade 요청으로 시작한다.  
핸드셰이크 요청은 `GET` 요청 라인 뒤에 여러 헤더가 무순서로 포함될 수 있다( RFC 2616 준수). 주요 헤더는 다음과 같다.

| 헤더 | 역할 |
|------|------|
| `Host` | 대상 서버 호스트명 지정 |
| `Upgrade: websocket` | 프로토콜 전환 의도 선언 |
| `Connection: Upgrade` | 연결 유지 방식 지정 |
| `Sec-WebSocket-Key` | 클라이언트 난수, 서버가 `Sec-WebSocket-Accept`를 계산하는 데 사용 |
| `Origin` | 브라우저 기반 보안 모델에 따라 출처 검증 |
| `Sec-WebSocket-Protocol` | 선택적 서브프로토콜 목록 |
| `Sec-WebSocket-Version` | 사용 버전(예: 13) |

서버는 클라이언트의 요청을 검증한 뒤 **101 Switching Protocols** 응답을 반환한다. 응답 헤더는 최소 `Upgrade`, `Connection`, `Sec-WebSocket-Accept`를 포함하고, 협상된 서브프로토콜이 있으면 `Sec-WebSocket-Protocol`을 추가한다.

핸드셰이크가 성공하면 양쪽은 **데이터 전송 단계**로 전환한다. 이 시점부터는 HTTP 형식이 사라지고, 프레임 기반의 바이너리 스트림이 교환된다.

## 프레이밍 개요

데이터 전송은 **프레임** 단위로 이루어진다. 하나의 논리적 메시지는 하나 이상의 프레임으로 구성될 수 있다. 프레임은 다음과 같은 유형을 가진다(총 6가지 정의, 10가지 예약).

| 프레임 타입 | 용도 |
|------------|------|
| 텍스트 프레임 | UTF‑8 텍스트 전송 |
| 바이너리 프레임 | 애플리케이션 정의 바이너리 데이터 |
| 컨트롤 프레임 | 프로토콜 제어(예: 연결 종료, Ping/Pong) |
| (기타 3가지) | RFC 6455에서 정의된 나머지 타입(예: 연속 프레임) |
| 예약(10개) | 미래 확장을 위해 비워둠 |

프레임은 **헤더 비트**와 **페이로드**로 구성된다. 클라이언트→서버 방향에서는 페이로드가 반드시 **마스킹**되어 전송된다(보안 모델상의 요구). 서버→클라이언트는 마스킹 없이 전송한다.

프레임 구조와 전송 순서는 RFC 6455 §5.1‑5.6에 정의되어 있으며, 프레임이 연속될 경우 동일 메시지에 속한다는 점을 기억한다.

## 설계 목표와 제한점

- **HTTP와의 호환성**: 핸드셰이크가 HTTP Upgrade 요청이므로 기존 프록시·로드밸런서와 동일 포트(80/443)에서 동작한다.  
- **보안 모델**: Origin 기반 검증과 클라이언트 마스킹을 통해 브라우저 환경에서 실행되는 불신 코드의 위험을 완화한다.  
- **확장성**: 6개의 정의된 프레임 타입 외에 10개의 예약 비트를 두어 향후 프로토콜 확장을 허용한다.  
- **복잡성**: 기존 HTTP 인프라를 재사용하면서도 양방향 전용 통신을 제공하기 위해 핸드셰이크와 프레이밍 로직이 추가된다(§1.2).

## 정리
WebSocket은 HTTP Upgrade 요청으로 시작되는 **핸드셰이크**와, **프레임 기반** 데이터 전송으로 구성된다. 핸드셰이크는 주요 헤더(`Upgrade`, `Connection`, `Sec-WebSocket-Key` 등)로 이루어지고, 성공 시 101 응답으로 전환된다. 프레임은 텍스트·바이너리·컨트롤 등 6가지 타입을 정의하며, 클라이언트→서버는 마스킹을 강제한다. 이러한 설계는 기존 HTTP 인프라와의 호환성을 유지하면서, 효율적인 양방향 통신을 제공한다.
