> 한 줄 요약: WebSocket은 HTTP 업그레이드 핸드셰이크로 연결을 설정한 뒤, TCP 위에서 양방향 메시지 프레이밍을 제공하는 프로토콜이다.

### 개요

WebSocket 프로토콜(RFC 6455)은 브라우저 기반 애플리케이션이 서버와 양방향 통신을 할 수 있게 하기 위해 설계되었다. 기존 HTTP 폴링 방식의 비효율성(다중 TCP 연결, 높은 헤더 오버헤드, 연결 매핑 복잡도)을 해결하고자 단일 TCP 연결 위에서 전이중 통신을 제공한다. 프로토콜은 **오프닝 핸드셰이크**와 **데이터 전송(프레이밍)** 두 부분으로 구성되며, 오리진 기반 보안 모델을 적용해 신뢰할 수 없는 클라이언트 코드로부터 서버를 보호한다.

## 배경: HTTP 폴링의 한계

전통적으로 웹에서 양방향 통신을 구현하려면 HTTP를 남용해야 했다. 클라이언트는 주기적으로 서버를 폴링해 업데이트를 확인하고, 업스트림 알림은 별도 HTTP 호출로 전송했다. 이 방식은 세 가지 주요 문제를 야기했다.

- 서버가 클라이언트당 여러 TCP 연결을 유지해야 함(하나는 다운스트림, 나머지는 업스트림 메시지용)
- 클라이언트→서버 메시지마다 HTTP 헤더가 붙어 와이어 프로토콜 오버헤드가 큼
- 클라이언트 스크립트가 나가는 연결과 들어오는 연결을 매핑해 응답을 추적해야 함

WebSocket은 단일 TCP 연결로 양방향 트래픽을 처리해 이 문제들을 해결한다. HTTP 포트(80, 443)와 프록시/중개자 인프라 위에서 동작하도록 설계돼 기존 인프라와의 호환성을 확보했다.

## 프로토콜 구조: 핸드셰이크와 데이터 전송

프로토콜은 두 단계로 나뉜다.

1. **오프닝 핸드셰이크**: HTTP `Upgrade` 요청/응답으로 WebSocket 연결을 협상
2. **데이터 전송**: 핸드셰이크 성공 후 프레임 단위로 메시지 교환

### 오프닝 핸드셰이크 상세

클라이언트 핸드셰이크는 표준 HTTP `GET` 요청에 `Upgrade: websocket` 헤더를 추가한 형태다. 주요 헤더 필드는 다음과 같다.

| 헤더 | 역할 |
|------|------|
| `Host` | 대상 호스트명 명시 (RFC 2616 준수) |
| `Upgrade: websocket` | 프로토콜 업그레이드 요청 |
| `Connection: Upgrade` | 연결 업그레이드 지시 |
| `Sec-WebSocket-Key` | 클라이언트 생성 난수(Base64 인코딩), 서버 검증용 |
| `Origin` | 요청 오리진, 동일 오리진 정책 적용 |
| `Sec-WebSocket-Protocol` | 클라이언트 지원 서브프로토콜 목록 (선택) |
| `Sec-WebSocket-Version` | 프로토콜 버전 (현재 13) |

서버는 요청을 검증한 뒤 `HTTP/1.1 101 Switching Protocols` 응답으로 프로토콜을 전환한다. 응답에는 `Sec-WebSocket-Accept` 헤더가 포함되는데, 이는 클라이언트 키에 고정 GUID(`258EAFA5-E914-47DA-95CA-C5AB0DC85B11`)를 붙여 SHA-1 해시 후 Base64 인코딩한 값이다. 이를 통해 클라이언트는 의도한 서버와 연결됐음을 확인한다.

헤더 순서는 무관하며, 쿠키 등 추가 헤더도 허용된다. `Request-URI`는 WebSocket 엔드포인트를 식별해 단일 IP/서버에서 다중 도메인/엔드포인트 운영을 가능하게 한다.

### 데이터 프레이밍 모델

핸드셰이크 완료 후 데이터는 **메시지** 단위로 교환되며, 각 메시지는 하나 이상의 **프레임**으로 구성된다. 프레임은 프로토콜 레벨의 최소 전송 단위다.

**프레임 타입 분류**
- **데이터 프레임**: 애플리케이션 페이로드 전달
  - 텍스트 프레임: UTF-8 인코딩 텍스트 (오피코드 `0x1`)
  - 바이너리 프레임: 해석은 애플리케이션 몫 (오피코드 `0x2`)
- **컨트롤 프레임**: 프로토콜 레벨 신호용
  - Close (`0x8`), Ping (`0x9`), Pong (`0xA`)
- **예약 오피코드**: 10개 향후 확장용 예약



**단편화(Fragmentation)**: 큰 메시지는 여러 프레임으로 분할 전송 가능하다. 첫 프레임은 실제 오피코드(텍스트/바이너리)를 갖고 `FIN=0`, 중간 연속 프레임은 오피코드 `0x0`에 `FIN=0`, 마지막 프레임은 오피코드 `0x0`에 `FIN=1`로 표시한다. 컨트롤 프레임은 단편화할 수 없으며(페이로드 ≤ 125바이트), 데이터 프레임 사이에 끼어들 수 있다.

**마스킹**: 클라이언트→서버 방향 모든 프레임은 32비트 마스킹 키로 XOR 마스킹된다. 이는 투명 프록시 등 인프라가 WebSocket 트래픽을 일반 HTTP로 오인해 캐싱/변조하는 공격을 방지하기 위함이다. 서버→클라이언트 방향은 마스킹하지 않는다.

## 보안 모델: 오리진 기반

WebSocket은 브라우저의 **오리진 기반 보안 모델**을 따른다. 클라이언트는 핸드셰이크에 `Origin` 헤더를 포함해 자신이 실행된 오리진을 알리고, 서버는 이 오리진을 검증해 연결을 수락할지 결정한다. 이는 신뢰할 수 없는 코드(예: 악성 웹페이지)가 임의 서버와 WebSocket 연결을 맺는 것을 방지한다. 비브라우저 클라이언트는 오리진 헤더를 위조할 수 있으므로 서버는 추가 인증 수단을 병행해야 한다.

## HTTP/TCP와의 관계

- **TCP 위 계층**: WebSocket은 TCP 스트림 위에서 프레이밍을 제공한다. TCP 연결이 끊기면 WebSocket 연결도 종료된다.
- **HTTP 호환성**: 핸드셰이크는 HTTP `Upgrade` 메커니즘을 사용해 단일 포트에서 HTTP와 WebSocket을 동시에 서비스할 수 있게 한다. 프록시/중개자 통과를 고려해 설계됐다.
- **전용 포트 가능성**: 명세는 HTTP에 종속되지 않으며, 향후 전용 포트에서 더 간단한 핸드셰이크로 동작할 여지를 남겨둔다.

## 연결 종료: 클로징 핸드셰이크

연결 종료는 **클로징 핸드셰이크**로 수행된다. 한쪽이 Close 컨트롤 프레임(상태 코드 + 선택적 사유 문자열)을 보내면, 상대방도 Close 프레임으로 응답한 뒤 TCP 연결을 닫는다. 상태 코드는 정상 종료(1000), 프로토콜 오류(1002), 데이터 타입 불일치(1003) 등 표준 코드와 예약 범위(1004-2999, 3000-3999, 4000-4999)로 정의된다. 비정상 종료(TCP 리셋, 타임아웃 등) 시 별도 복구 절차가 정의돼 있다.

## 확장성과 서브프로토콜

- **확장(Extensions)**: 핸드셰이크 단계에서 `Sec-WebSocket-Extensions` 헤더로 협상. 압축(permessage-deflate) 등이 대표적.
- **서브프로토콜(Subprotocols)**: `Sec-WebSocket-Protocol` 헤더로 애플리케이션 레벨 프로토콜(예: `chat`, `mqtt`)을 협상. 서버는 클라이언트 목록 중 하나만 선택해 응답.

## 트레이드오프와 주의사항

- **SHA-1 사용**: 핸드셰이크 검증에 SHA-1을 쓰는데, 충돌 공격 이론적 가능성은 있으나 프리이미지 저항성은 유지돼 현재 용도엔 충분하다고 판단됨.
- **마스킹 오버헤드**: 클라이언트→서버 프레임마다 4바이트 마스킹 키 + XOR 연산 비용 발생. 보안(프록시 공격 방지)과 성능 간 트레이드오프.
- **최대 프레임 크기**: 컨트롤 프레임은 125바이트로 제한. 데이터 프레임은 64비트 길이 필드 지원(최대 2^63-1 바이트). 구현체별 제한 다를 수 있음.
- **UTF-8 검증**: 텍스트 프레임 수신 시 유효한 UTF-8인지 검증해야 하며, 위반 시 연결 종료(코드 1007) 필요.
- **인프라 호환성 복잡도**: HTTP 프록시/중개자 통과를 위해 핸드셰이크가 HTTP 형태를 띠지만, 이로 인해 프로토콜 복잡도 증가.

## 정리

- WebSocket은 HTTP 폴링의 비효율을 해소하고자 단일 TCP 연결 위에서 전이중 통신을 제공하는 프로토콜이다.
- 연결은 HTTP `Upgrade` 핸드셰이크로 시작하며, `Sec-WebSocket-Key`/`Accept` 쌍으로 서버 검증, `Origin` 헤더로 오리진 기반 보안 적용.
- 데이터는 메시지 단위로 교환되며, 메시지는 하나 이상의 프레임(텍스트/바이너리/컨트롤)으로 구성된다. 단편화와 클라이언트→서버 마스킹이 핵심 프레이밍 특징이다.
- 컨트롤 프레임(Close/Ping/Pong)으로 연결 생명주기와 킵얼라이브를 관리하며, 확장/서브프로토콜 협상으로 기능 확장이 가능하다.
- 설계는 기존 HTTP 인프라(프록시, 포트 80/443) 호환성을 우선하되, 향후 전용 포트 단순화 여지를 남겨둔다.
