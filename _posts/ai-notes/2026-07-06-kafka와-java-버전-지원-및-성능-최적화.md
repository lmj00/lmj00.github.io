### 개요
Apache Kafka는 JVM 위에서 동작하는 분산 시스템으로, Java 버전 선택은 성능, 안정성, 보안에 직접적인 영향을 미친다. 공식 문서는 지원되는 Java 버전의 범위와 각 버전의 지원 수준을 명확히 정의하며, 실제 대규모 운영 환경에서 검증된 성능 최적화 지침을 제시한다. 이는 운영자가 올바른 Java 런타임을 선택하고, Kafka 브로커의 가비지 컬렉션(GC) 성능을 튜닝하는 데 핵심적인 기준이 된다.

### 지원되는 Java 버전
Kafka는 여러 Java 버전에 대해 **계층적 지원 정책**을 따른다.

**완전 지원(Full Support)** 되는 버전은 Java 17, Java 21, Java 25이다. 이는 Kafka의 모든 모듈(브로커, 클라이언트, 커넥트, 스트림즈 등)이 해당 Java 버전에서 정상적으로 동작함을 의미한다.

**제한적 지원(Limited Support)** 은 Java 11에 적용된다. Java 11은 클라이언트(`clients`), 스트림즈(`streams`) 및 관련 모듈에서만 지원된다. 브로커와 같은 서버 측 모듈에 대한 지원은 포함되지 않을 수 있다.

**최신 비LTS 버전**에 대한 지원은 **최선의 노력(best-effort)** 기반이다. 프로젝트는 일반적으로 가장 최근의 비LTS(장기 지원 버전이 아닌) 버전으로만 테스트를 수행한다. 따라서 최신 LTS 버전보다 더 새로운 Java 버전을 사용할 경우, 공식적인 지원 보장보다는 커뮤니티의 기여와 호환성 테스트에 의존하게 된다.

### 버전 선택 권장사항
공식 문서는 다음과 같은 기준으로 Java 버전을 선택할 것을 권장한다.

1.  **성능과 효율성, 지원 수준**: **가장 최근의 LTS 릴리스**를 사용하는 것이 좋다. 문서 작성 시점에서는 Java 25가 해당된다. LTS 버전은 장기간 안정적인 업데이트와 성능 개선이 보장된다.
2.  **보안 관점**: **최신 패치 버전**을 사용할 것을 강력히 권장한다. 오래된 패치 버전은 일반적으로 공개된 보안 취약점을 포함하고 있기 때문이다.

이 권장사항은 Kafka의 안정적인 운영과 최적의 성능을 보장하기 위한 기본 원칙이다.

### JVM 성능 최적화 인수
문서는 OpenJDK 기반의 Java 구현체(오라클 JDK 포함)로 Kafka를 실행할 때 사용할 수 있는 **전형적인 JVM 인수 집합**을 제시한다. 이 설정은 대용량 트래픽을 처리하는 실제 운영 환경(LinkedIn의 예시)에서 검증되었다.

주요 인수와 그 목적은 다음과 같다.
*   힙 메모리: `-Xmx6g -Xms6g`으로 초기 및 최대 힙 크기를 6GB로 고정하여 GC 오버헤드를 줄인다.
*   가비지 컬렉터: `-XX:+UseG1GC`로 G1 가비지 컬렉터를 지정한다. G1은 큰 힙에서 예측 가능한 일시 정지 시간을 제공하는 데 적합하다.
*   GC 성능 목표: `-XX:MaxGCPauseMillis=20`은 GC 일시 정지 시간 목표를 20ms로 설정하고, `-XX:InitiatingHeapOccupancyPercent=35`는 힙 점유율이 35%에 도달하면 GC를 시작하도록 한다.
*   메타스페이스 튜닝: `-XX:MetaspaceSize=96m`으로 초기 크기를 설정하고, `-XX:MinMetaspaceFreeRatio=50 -XX:MaxMetaspaceFreeRatio=80`으로 여유 공간 비율을 관리한다.
*   명시적 GC 처리: `-XX:+ExplicitGCInvokesConcurrent`는 시스템 호출에 의한 명시적 GC(`System.gc()`)도 병렬로 처리되도록 한다.

### 검증된 운영 사례
제시된 JVM 인수 설정의 효용성은 LinkedIn의 가장 바쁜 클러스터 운영 통계로 뒷받침된다.
*   규모: 60대의 브로커, 5만 개의 파티션(복제 계수 2).
*   처리량: 초당 80만 메시지 수신, 초당 300MB 인바운드, 초당 1GB 이상의 아웃바운드 트래픽.
*   GC 성능: 해당 클러스터의 모든 브로커는 **90%의 GC 일시 정지 시간이 약 21ms**에 달하며, 초당 1회 미만의 Young GC만 발생한다.

이 데이터는 제안된 설정이 매우 높은 부하 하에서도 예측 가능하고 짧은 GC 일시 정지를 유지할 수 있음을 보여준다.

### 고려사항 및 주의점
*   **Java 11 사용 제한**: Java 11을 브로커에 사용하려는 경우 공식 지원 범위를 반드시 확인해야 한다. 문서는 클라이언트 및 스트림즈에 대한 지원만을 명시하고 있다.
*   **비LTS 버전의 위험성**: 최신 비LTS Java 버전은 새로운 기능을 빨리 사용할 수 있지만, 지원이 보장되지 않으며 예기치 않은 호환성 문제가 발생할 수 있다.
*   **하드웨어 의존성**: 제시된 JVM 인수(예: 6GB 힙)는 특정 규모의 운영 사례를 기반으로 한 참고 자료이다. 실제 환경의 메모리 크기, CPU 코어 수, 워크로드 패턴에 따라 튜닝이 필요하다.

### 정리
Kafka의 Java 버전 지원 정책은 완전 지원(LTS 버전군), 제한적 지원(Java 11), 최선의 노력 지원(최신 비LTS)으로 구분된다. 운영 시에는 보안과 장기 지원을 고려해 최신 LTS 버전 사용이 원칙이다. 성능 최적화를 위해서는 G1 GC를 중심으로 한 JVM 인수 튜닝이 효과적이며, LinkedIn의 대규모 클러스터 사례는 이러한 설정이 실전에서 검증되었음을 증명한다. 환경에 맞는 Java 버전 선택과 꾸준한 모니터링 기반의 JVM 튜닝이 Kafka 클러스터의 안정적 운영을 위한 핵심이다.
