### 개요
컨테이너 이미지는 애플리케이션과 모든 소프트웨어 의존성을 캡슐화한 **실행 가능한 소프트웨어 번들**이다. 쿠버네티스에서 파드를 정의할 때는 보통 레지스트리에 푸시된 이미지를 참조한다. 이 개념 노트는 쿠버네티스가 이미지 이름을 어떻게 해석하고, 어떤 정책으로 이미지를 가져오며(**풀링**), 병렬 처리를 어떻게 제어하는지 공식 문서를 바탕으로 설명한다.

### 이미지 이름과 식별자
이미지 이름은 `pause`, `example/mycontainer`, `kube-apiserver`와 같은 형식이다. **레지스트리 호스트명**(예: `fictional.registry.example/imagename`)과 **포트 번호**(예: `:10443`)를 포함할 수 있다. 호스트명을 지정하지 않으면 쿠버네티스는 **Docker 공개 레지스트리**(`docker.io`)를 사용한다고 가정한다.

이미지의 특정 버전을 지정하는 방법은 두 가지다.
*   **태그(Tag)**: 동일 이미지 시리즈의 다른 버전을 식별한다. 소문자, 대문자, 숫자, 밑줄(`_`), 마침표(`.`), 대시(`-`)로 구성되며 최대 128자까지 가능하다. 태그를 명시하지 않으면 쿠버네티스는 `latest` 태그를 사용한다.
*   **다이제스트(Digest)**: 이미지 콘텐츠의 해시 값(예: `sha256:1ff6c18fbef...`)으로, **변경 불가능한 고유 식별자**다. 태그는 다른 이미지를 가리키도록 변경될 수 있지만, 다이제스트는 특정 버전을 고정한다.

다음은 쿠버네티스에서 사용 가능한 이미지 이름 예시다.

| 예시 | 설명 |
| :--- | :--- |
| `busybox` | 태그/다이제스트 없음. `docker.io/library/busybox:latest`로 해석. |
| `busybox:1.32.0` | 태그 지정. `docker.io/library/busybox:1.32.0`으로 해석. |
| `registry.k8s.io/pause:3.5` | 커스텀 레지스트리와 태그 지정. |
| `registry.k8s.io/pause@sha256:1ff6c18f...` | 다이제스트로 이미지 버전 고정. |
| `registry.k8s.io/pause:3.5@sha256:1ff6c18f...` | 태그와 다이제스트 모두 지정. 풀링 시에는 **다이제스트만 사용**. |

**프로덕션 환경에서는 `:latest` 태그 사용을 피하고, 의미 있는 태그(예: `v1.42.0`)나 다이제스트를 지정하는 것이 권장된다.** 이렇게 하면 실행 중인 이미지 버전을 추적하고 롤백하기가 쉬워진다. 다이제스트를 사용하면 레지스트리에서 태그가 가리키는 코드가 변경되더라도 쿠버네티스가 항상 동일한 버전의 이미지를 실행하게 되어 파드 간 버전 불일치를 방지할 수 있다.

### 이미지 풀 정책 (imagePullPolicy)
컨테이너의 `imagePullPolicy`와 이미지 태그는 **kubelet이 이미지를 가져오는 시점**을 결정한다. 파드 템플릿을 처음 생성할 때 풀 정책을 명시하지 않으면 기본값이 `IfNotPresent`로 설정된다.

다음은 `imagePullPolicy`의 세 가지 값과 그 효과다.

| 값 | 동작 |
| :--- | :--- |
| `IfNotPresent` | 이미지가 로컬에 **없을 때만** 풀링한다. |
| `Always` | 컨테이너를 시작할 때마다 컨테이너 런타임에 풀링을 요청한다. 컨테이너 런타임은 레지스트리에 연락해 태그를 다이제스트로 확인하고, 로컬에 캐시되지 않은 레이어만 다운로드한다. 모든 레이어가 캐시되어 있으면 다시 다운로드하지 않는다. |
| `Never` | 이미지를 가져오지 않는다. 이미지가 로컬에 존재해야만 컨테이너를 시작할 수 있다. |

`Always` 정책도 컨테이너 런타임의 캐싱 덕분에 레이어가 존재하면 다시 다운로드하지 않으므로 효율적이다.

**기본 풀 정책 설정 규칙**은 다음과 같다. 이 규칙은 객체(파드 등)가 **처음 생성될 때 한 번 적용**되며, 이후 이미지 태그나 다이제스트가 변경되어도 자동으로 업데이트되지 않는다.
*   `imagePullPolicy`를 생략하고 **다이제스트**를 지정 → `IfNotPresent`
*   `imagePullPolicy`를 생략하고 태그가 **`:latest`** → `Always`
*   `imagePullPolicy`를 생략하고 태그를 **전혀 지정하지 않음** → `Always`
*   `imagePullPolicy`를 생략하고 `:latest`가 **아닌 태그**를 지정 → `IfNotPresent`

항상 강제 풀링을 원한다면 `imagePullPolicy`를 `Always`로 설정하거나, 태그를 `:latest`로 지정(또는 생략)하여 쿠버네티스가 `Always`로 설정하게 하면 된다.

### 이미지 풀링 메커니즘과 제어
**ImagePullBackOff**는 컨테이너 이미지를 풀링할 수 없을 때(잘못된 이미지 이름, 비공개 레지스트리 인증 실패 등) 컨테이너가 `Waiting` 상태가 되며 발생하는 상태다. 쿠버네티스는 지수 백오프 방식으로 풀링을 재시도하며, 지연 시간은 최대 300초(5분)까지 증가한다.

기본적으로 kubelet은 **직렬 이미지 풀링**을 수행한다. 즉, 한 번에 하나의 이미지 풀 요청만 이미지 서비스에 보낸다. `serializeImagePulls`를 `false`로 설정하면 **병렬 이미지 풀링**이 활성화되어 여러 이미지가 동시에 풀링된다. 단, **한 파드 내의 여러 컨테이너(예: init 컨테이너와 앱 컨테이너) 이미지는 병렬로 풀링되지 않는다.** 서로 다른 파드의 이미지는 병렬로 풀링될 수 있다.

병렬 풀링을 활성화한 경우, `maxParallelImagePulls` 필드를 설정하여 동시에 풀링할 수 있는 이미지 수의 **최대값을 제한**할 수 있다. 이는 네트워크 대역폭이나 디스크 I/O를 과도하게 소모하는 것을 방지한다. `maxParallelImagePulls`를 2 이상으로 설정하려면 반드시 `serializeImagePulls`를 `false`로 설정해야 한다.

### 비공개 레지스트리 사용
비공개 레지스트리에서 이미지를 풀링하려면 인증이 필요하다. 주요 인증 방법은 다음과 같다.
1.  **Pod에 imagePullSecrets 지정 (권장)**: 파드 스펙에 `imagePullSecrets` 필드를 설정한다. 이 시크릿은 파드와 동일한 네임스펙스에 존재해야 하며, `kubernetes.io/dockercfg` 또는 `kubernetes.io/dockerconfigjson` 타입이어야 한다.
2.  **노드 인증 구성**: 모든 노드를 비공개 레지스트리에 인증하도록 구성한다. 클러스터 관리자가 수행해야 하며, 해당 노드의 모든 파드가 레지스트리에 접근할 수 있게 된다.
3.  **kubelet 자격 증명 공급자**: kubelet이 플러그인을 실행해 동적으로 레지스트리 자격 증명을 가져오도록 구성한다. **정적 파드(Static Pod)**가 비공개 레지스트리 이미지를 사용해야 할 때 특히 유용하다.

`config.json` 파일의 `auths` 항목에서 Kubernetes는 Docker와 달리 **글로브 패턴(`*`)을 포함한 URL**을 지원한다. 패턴은 각 서브도메인에 대해 점(`.`)을 포함해야 하며, 매칭되는 서브도메인 수와 패턴 수가 같아야 한다(예: `*.my-registry.example/images`는 `sub.my-registry.example/images`와 매치되지만, `my-registry.example`이나 `a.b.sub.my-registry.example/images`와는 매치되지 않는다). kubelet은 매칭되는 모든 자격 증명 패턴에 대해 순차적으로 이미지 풀을 시도한다.

### 정리
*   컨테이너 이미지는 태그(가변적)나 다이제스트(고정적)로 버전을 지정하며, 프로덕션에서는 다이제스트 사용이 버전 관리에 유리하다.
*   `imagePullPolicy`(`Always`, `IfNotPresent`, `Never`)는 이미지가 로컬에 있을 때 풀링을 생략할지 여부를 결정하며, 객체 생성 시 이미지 식별자에 따라 기본값이 자동 설정된다.
*   kubelet은 기본적으로 이미지를 직렬로 풀링하지만, 병렬 풀링과 그 동시 실행 수를 제한하도록 구성할 수 있다.
*   비공개 레지스트리 사용 시 파드 수준의 `imagePullSecrets` 지정이 권장되며, kubelet 자격 증명 공급자는 정적 파드 시나리오에 적합하다.
