> 한 줄 요약: ABNF(Augmented BNF)는 인터넷 기술 명세서에서 공식 문법을 정의하기 위해 널리 사용되는 메타 언어다. 기존 BNF에 비해 규칙 이름, 반복, 대안, 값 범위 등에서 향상된 표현력을 제공하며, 간결성과 단순성, 합리적인 표현력을 균형 있게 갖추고 있다.

### 개요
인터넷 기술 명세서는 공식적인 문법을 정의해야 하는 경우가 많다. 역사적으로 **BNF(Backus-Naur Form)** 의 수정된 버전인 **ABNF(Augmented BNF)** 가 많은 인터넷 명세서에서 인기를 얻어 왔다. RFC 5234는 이 ABNF를 표준으로 문서화한다. ABNF는 초기 아르파넷 시절 각 명세서가 자체적으로 ABNF를 정의하던 관행을 통합하여 선택적 참조를 가능하게 하며, 몇 가지 수정과 향상된 기능을 제공한다.

### 규칙 정의의 기본 구조
ABNF에서 문법은 **규칙(rule)** 의 집합으로 정의된다. 각 규칙은 다음 형식을 따른다.
```
name = elements crlf
```
여기서 `name`은 규칙 이름, `elements`는 하나 이상의 규칙 이름이나 터미널 값 명세이며, `crlf`는 줄바꿈(CR LF)을 나타낸다. 등호(`=`)는 이름과 규칙 정의를 구분한다.

**규칙 이름**은 알파벳 문자로 시작하고, 알파벳, 숫자, 하이픈의 조합으로 이어진다. 규칙 이름은 **대소문자를 구분하지 않는다**. 원래 BNF와 달리 꺾쇠괄호(`<`, `>`)는 필수가 아니지만, 규칙 이름을 구별하기 쉽게 할 때 사용할 수 있다.

규칙 정의는 왼쪽 정렬되며, 여러 줄이 필요한 경우 연속 줄은 들여쓰기된다.

### 터미널 값과 외부 인코딩
규칙은 최종적으로 **터미널 값**(문자)의 문자열로 해석된다. ABNF에서 문자는 단순히 음이 아닌 정수이며, 특정 컨텍스트에서 ASCII와 같은 문자 집합으로의 매핑(인코딩)이 지정될 수 있다.

터미널 값은 명시적으로 기수를 표시하여 하나 이상의 숫자 문자로 지정된다.
* `%b`: 2진수
* `%d`: 10진수
* `%x`: 16진수

예를 들어 캐리지 리턴은 `CR = %d13` 또는 `CR = %x0D`로 정의할 수 있다. 점(`.`)을 사용하여 값을 연결할 수 있다(`CRLF = %d13.10`).

**따옴표로 둘러싼 리터럴 문자열**도 직접 지정할 수 있다(`command = "command string"`). ABNF 문자열은 **대소문자를 구분하지 않으며**, 문자 집합은 US 7비트 ASCII를 가정한다. 대소문자를 구분하는 문자열을 정의하려면 각 문자를 개별 숫자 값으로 지정해야 한다(`rulename = %d97 %d98 %d99`).

터미널 값의 외부 표현(인코딩)은 저장 또는 전송 환경의 제약에 따라 달라질 수 있다. 따라서 동일한 ABNF 문법이 7비트 US-ASCII, 바이너리 옥텟, 16비트 유니코드 등 여러 외부 인코딩을 가질 수 있다. 인코딩 세부 사항은 ABNF의 범위를 벗어나며, 부록 B에서는 인터넷에서 흔히 사용되는 7비트 US-ASCII 환경에 대한 핵심 규칙 정의를 편의상 제공한다.

### ABNF 연산자
ABNF는 문법을 구성하기 위한 여러 연산자를 정의한다. 다음 표는 주요 연산자와 그 의미를 정리한 것이다.

| 연산자 | 의미 | 예시 및 설명 |
| :--- | :--- | :--- |
| `Rule1 Rule2` | **연결(Concatenation)** | `foo bar`는 `foo` 다음에 `bar`가 오는 순서를 정의한다. ABNF는 선형 공백(스페이스, 탭)을 암시적으로 허용하지 않으므로, 필요시 명시적으로 정의해야 한다. |
| `Rule1 / Rule2` | **대안(Alternatives)** | `foo / bar`는 `foo` 또는 `bar`를 의미한다. |
| `Rule1 =/ Rule2` | **증분 대안(Incremental Alternatives)** | 기존 규칙에 새로운 대안을 추가할 수 있다. `oldrule =/ newalt` |
| `%c##-##` | **값 범위 대안(Value Range Alternatives)** | `DIGIT = %x30 39`는 `"0" / "1" / ... / "9"`와 동일하다. 연결된 값(점)과 값 범위(대시)는 동일한 문자열에서 함께 사용할 수 없다. |
| `(Rule1 Rule2)` | **시퀀스 그룹(Sequence Group)** | 괄호 안의 요소들은 단일 요소로 취급되어 엄격한 순서를 가진다. 그룹화는 모호함을 피하기 위해 권장된다. |
| `*Rule`<br>`<a>*<b>element` | **가변 반복(Variable Repetition)** | `*element`는 0회 이상, `1*element`는 1회 이상, `3*3element`는 정확히 3회, `1*2element`는 1회 또는 2회 반복을 의미한다. |
| `nRule` | **특정 반복(Specific Repetition)** | `2DIGIT`는 정확히 두 자리 숫자를 의미하며, `2*2DIGIT`와 동일하다. |
| `[RULE]` | **선택적 시퀀스(Optional Sequence)** | `[foo bar]`는 `*1(foo bar)`와 동일하며, 0회 또는 1회 나타날 수 있음을 의미한다. |
| `; Comment` | **주석(Comment)** | 세미콜론부터 줄 끝까지가 주석이다. |

### 연산자 우선순위
여러 연산자가 혼용될 때는 다음 우선순위가 적용된다(위에서 아래로 우선순위 높음 → 낮음).
1. 규칙 이름, 설명 값, 터미널 값
2. 주석
3. 값 범위
4. 반복
5. 그룹화, 선택적
6. 연결
7. 대안

연결과 대안 연산자가 자유롭게 혼합되면 혼란스러울 수 있으므로, **그룹화 연산자를 사용하여 명시적인 연결 그룹을 만드는 것이 권장된다.**

### ABNF의 설계 특징과 한계
ABNF는 **표현력과 간결성의 균형**을 목표로 설계되었다. 표준 BNF와의 주요 차이점은 규칙 이름(대소문자 무시, 꺾쇠괄호 불필요), 반복 표기법(`*`, `n`), 대안 표기법(`/`, `=/`), 순서 독립성, 값 범위(`%c##-##`) 지원에 있다.

ABNF는 문법 정의와 **외부 인코딩을 분리**하여, 동일한 문법을 다양한 인코딩 환경(7비트 ASCII, 바이너리, 유니코드)에서 사용할 수 있도록 의도했다. 이는 구현 유연성을 높이는 장점이다.

그러나 ABNF 문법 자체는 상대적으로 **엄격한 형식**을 요구한다. 또한 선형 공백을 암시적으로 허용하지 않기 때문에, 문법에서 공백을 허용하려면 명시적으로 핵심 규칙으로 정의해야 한다는 점에 유의해야 한다.

### 정리
* ABNF는 인터넷 프로토콜(RFC) 명세서에서 문법을 정의하는 데 사실상 표준으로 사용되는 메타 언어다.
* 규칙 정의는 `이름 = 요소` 형식을 따르며, 규칙 이름은 대소문자를 구분하지 않는다.
* 터미널 값은 숫자(`%d`, `%x`, `%b`) 또는 리터럴 문자열로 지정되며, 리터럴 문자열은 기본적으로 대소문자를 구분하지 않는다.
* 연결(` `), 대안(`/`), 반복(`*`, `n`), 그룹화(`()`), 선택(`[]`) 등의 연산자를 조합하여 복잡한 문법을 표현할 수 있다.
* 문법 정의와 물리적 인코딩을 분리함으로써 다양한 환경에서의 적용성을 확보했다.
