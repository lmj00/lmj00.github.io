> 한 줄 요약: Kubernetes API 서버가 객체를 저장할 때 사용하는 **스토리지 버전**의 개념과, CustomResourceDefinition(CRD)에서 여러 API 버전을 정의하고 그 중 하나를 저장용 버전으로 지정하는 방법을 설명한다.

### 개요
Kubernetes 클러스터에서 API 서버는 객체를 etcd와 같은 백엔드 저장소에 저장한다. 이때 객체는 특정 API 버전의 형식으로 직렬화되어 저장되는데, 이를 **스토리지 버전**이라고 부른다. 이 개념은 클러스터 관리자에게 중요한데, 객체의 API 표현과 저장소 내 실제 인코딩을 연결해주기 때문이다. 특히 휴지 상태 암호화나 API 사용 중단과 같은 시나리오에서 저장된 객체의 이진 인코딩이 중요해진다.

스토리지 버전은 클라이언트가 사용하는 API 버전과 완전히 별개다. API 서버는 자동 변환을 통해 클라이언트가 v1, v2 등 다양한 API 버전으로 객체와 상호작용할 수 있게 하며, 내부적으로는 저장된 버전을 필요한 API 표현으로 변환한다.

### 스토리지 버전의 동작 원리
하나의 API 리소스는 여러 스토리지 버전을 가질 수 있으며, API 서버는 이들 사이의 변환을 알고 있다. 그러나 특정 시점에 하나의 객체는 오직 **한 개의 스토리지 버전**으로만 저장된다. 모든 리소스는 언제든지 하나의 **활성 스토리지 버전**을 가지며, 객체에 대한 모든 쓰기 작업은 그 버전으로 저장된다.

읽기 요청 시 API 서버는 저장된 데이터를 객체의 API 표현으로 변환한다. 따라서 객체가 갱신되지 않는 한 오래된 스토리지 버전으로 저장된 객체는 무기한 유지될 수 있다. 반면 쓰기(갱신) 작업이 발생하면, 저장된 객체는 새로운 스토리지 버전 표현으로 변환되어 기록된다.

### CustomResourceDefinition(CRD)의 버전 관리
내장된 Kubernetes 타입과 달리, 동적으로 정의되는 **커스텀 리소스**는 스토리지 버전 관리 방식이 다르다. CRD를 정의할 때는 `versions` 필드 아래에 지원할 API 버전 목록을 나열해야 한다. 각 버전은 두 가지 중요한 플래그를 가진다.

| 플래그 | 설명 |
| :--- | :--- |
| `served` | 해당 API 버전을 클라이언트에 제공할지 여부. `true`로 설정하면 클라이언트가 이 버전으로 CRD 객체를 조회하거나 생성할 수 있다. |
| `storage` | 해당 버전을 **스토리지 버전**으로 사용할지 여부. 여러 버전 중 **정확히 하나의 버전만** `true`로 설정해야 한다. |

공식 문서의 `crontabs` 예제 CRD를 보면, `v1beta1`과 `v1` 두 버전이 모두 `served: true`로 설정되어 클라이언트가 사용할 수 있다. 그러나 `storage: true`는 `v1beta1`에만 설정되어 있다. 이는 다음과 같은 중요한 의미를 가진다.
*   모든 `CronTab` 객체의 생성 또는 갱신은 `v1beta1`의 스키마(`host`, `port` 필드만 정의됨)를 따라 저장소에 인코딩된다.
*   클라이언트가 `v1` API(여기서는 `time` 필드가 추가됨)를 사용해 객체를 생성하려 해도, 실제 저장되는 것은 `v1beta1` 스키마이므로 `time` 필드는 저장되지 않는다.
*   두 버전을 동시에 스토리지 버전으로 표시하는 것은 유효하지 않다. 그렇게 되면 객체를 저장하는 데 두 가지 데이터 스키마가 동시에 유효한 방식이 되어 버리기 때문이다.

### 스토리지 버전 변경과 관련 주의사항
스토리지 버전을 변경하면, 새로 생성되거나 갱신되는 커스텀 리소스(CR) 객체부터 새로운 버전이 사용된다. 기존 객체를 조회(`get`)하거나 감시(`watch`)하는 것은 변환만 일어날 뿐 객체 자체에 영향을 주지 않는다. 스토리지 버전 변경은 **갱신 또는 생성 작업을 통해서만 효과가 발생**한다.

이러한 메커니즘은 클러스터 관리자에게 몇 가지 문제를 제기한다.
1.  **API 버전 사용 중단의 어려움**: 오래된 스토리지 버전을 사용하는 객체가 더 이상 없을 때까지는 해당 API 버전을 안전하게 제거할 수 없다. 모든 객체가 새로운 스토리지 버전을 사용하는지 투명하게 확인하기 어려워, 버전 제거 시기를 판단하기 힘들다.
2.  **휴지 상태 암호화와의 연관성**: 저장소 암호화가 활성화된 경우, API 서버는 특정 스토리지 버전의 키를 사용해 객체를 복호화한다. 키 순환 시, 모든 객체가 최소 한 번은 쓰기 작업을 통해 새로운 스토리지 버전(및 새로운 암호화 키)으로 마이그레이션될 때까지는 **구키와 신키를 모두 사용 가능한 상태로 유지**해야 한다. 이는 보안 위험과 사용성 문제를 동시에 초래한다.

이러한 문제를 해결하기 위해 Kubernetes는 **스토리지 버전 마이그레이션** 도구를 제공하여, 수동 개입 없이 모든 객체가 새로운 스토리지 버전을 사용하도록 보장하는 마이그레이션을 실행할 수 있다.

### 정리
*   **스토리지 버전**은 객체가 실제 저장소에 직렬화되는 API 버전이다. 클라이언트가 사용하는 API 버전과 독립적이다.
*   CRD는 `versions` 목록을 통해 여러 API 버전을 지원할 수 있으며, 각 버전은 `served`(제공 여부)와 `storage`(저장 버전 여부) 플래그로 제어된다.
*   **정확히 하나의 버전만** `storage: true`로 설정되어 활성 스토리지 버전이 되어야 한다. 이 버전의 스키마가 저장소 계층의 객체 인코딩을 정의한다.
*   스토리지 버전 변경은 기존 객체를 자동으로 변환하지 않으며, 갱신 또는 생성 작업을 통해 점진적으로 적용된다. 이는 API 사용 중단 및 암호화 키 관리에 복잡성을 더한다.
