### 개요
쿠버네티스 API는 쿠버네티스 내 **객체(Pod, Namespace, ConfigMap, Event 등)의 상태를 질의하고 조작**할 수 있게 하는 HTTP API다. 이 API 서버는 컨트롤 플레인의 핵심으로, 사용자, 클러스터 내부 구성 요소, 외부 컴포넌트가 모두 이를 통해 서로 통신한다. 대부분의 작업은 `kubectl`이나 `kubeadm` 같은 CLI 도구를 통해 이루어지지만, REST 호출이나 클라이언트 라이브러리를 이용해 직접 접근할 수도 있다.

### API 서버와 통신 경로
모든 통신은 **API 서버**를 중심으로 이루어진다. API 서버는 HTTP API를 노출하여 클러스터 상태에 대한 유일한 진입점을 제공한다. 아래 다이어그램은 주요 구성 요소들이 API 서버를 통해 상호작용하는 방식을 보여준다.

### API 사양 공개: Discovery API와 OpenAPI
각 쿠버네티스 클러스터는 자신이 제공하는 API의 사양을 공개하며, 자동화된 상호운용성을 위해 두 가지 메커니즘이 사용된다. `kubectl`은 명령어 완성 등의 기능을 위해 이 API 사양을 가져와 캐시한다.

1.  **Discovery API**: 사용 가능한 **API 이름, 리소스, 버전, 지원 작업**에 대한 간략한 요약 정보를 제공한다. 리소스의 상세 스키마는 포함하지 않는다. 이 API는 **집계(aggregated)** 형태와 **비집계(unaggregated)** 형태로 제공된다.
2.  **Kubernetes OpenAPI 문서**: 모든 API 엔드포인트에 대한 **완전한 OpenAPI v2.0 및 v3.0 스키마**를 제공한다. OpenAPI v3이 더 포괄적이고 정확한 API 뷰를 제공하므로 선호되는 방법이다.

**집계 Discovery** (v1.30 안정화)는 클러스터가 지원하는 모든 리소스 정보를 `/api`와 `/apis` 두 개의 엔드포인트를 통해 게시한다. 이는 Discovery 데이터를 가져오기 위해 클러스터에 보내는 요청 수를 크게 줄인다. `Accept: application/json;v=v2;g=apidiscovery.k8s.io;as=APIGroupDiscoveryList` 헤더를 통해 이 리소스 타입을 요청해야 한다.

**비집계 Discovery**는 계층적으로 공개된다. 루트 엔드포인트(`/api`, `/apis`)에서 지원하는 모든 그룹 버전 목록을 게시하고, 각 그룹 버전별 Discovery 문서를 얻기 위해서는 `/apis/<group>/<version>`과 같은 추가 요청이 필요하다.

### OpenAPI v2와 v3의 특징
쿠버네티스는 OpenAPI v2.0과 v3.0을 모두 제공한다. OpenAPI v3은 리소스의 더 포괄적인(무손실) 표현을 제공하며, v2의 한계로 인해 `default`, `nullable`, `oneOf` 같은 일부 필드가 누락되는 문제를 해결한다.

**OpenAPI v3** (v1.27 안정화)은 `/openapi/v3` 엔드포인트를 통해 사용 가능한 모든 그룹/버전 목록을 JSON 형식으로 제공한다. 각 그룹 버전별 스펙은 `/openapi/v3/apis/<group>/<version>?hash=<hash>` 엔드포인트에서 제공되며, 해시값을 사용한 **불변 URL**을 통해 클라이언트 측 캐싱을 개선한다. API 서버는 이를 위해 `Expires`와 `Cache-Control: immutable` 같은 적절한 HTTP 캐싱 헤더를 설정한다.

OpenAPI 스키마의 일부로 게시된 유효성 검사 규칙은 완전하지 않을 수 있으며, 추가 검증이 API 서버 내부에서 발생한다. 정확하고 완전한 검증을 원한다면 `kubectl apply --dry-run=server`를 실행해야 한다.

### API 그룹과 버전 관리
확장성과 진화를 용이하게 하기 위해 쿠버네티스는 **API 그룹**을 구현하며, 이를 활성화하거나 비활성화할 수 있다. 리소스 표현을 재구성하거나 필드를 제거하기 쉽도록 **여러 API 버전**을 지원하며, 버전 관리는 리소스나 필드 수준이 아닌 **API 수준**에서 이루어진다. 이는 시스템 리소스와 동작에 대한 명확하고 일관된 뷰를 제공하고, 수명이 끝난 실험적 API에 대한 접근을 제어하기 위함이다.

API 서버는 여러 API 버전 간의 변환을 투명하게 처리한다. 모든 다른 버전은 동일한 영구 저장 데이터(`etcd`에 저장)의 표현일 뿐이다. 예를 들어, `v1beta1` API로 생성한 객체는 `v1` API가 사용 가능해진 후에도 두 버전을 통해 읽기, 업데이트, 삭제가 가능하다. `v1beta1` 버전이 제거되면 `v1` API를 통해 객체에 계속 접근할 수 있다.

### API 변경과 호환성 정책
쿠버네티스 API는 성장하고 변화하도록 설계되었다. 프로젝트는 기존 클라이언트와의 호환성을 깨지 않도록 유지하는 것을 목표로 한다.
*   **새 리소스/필드 추가**: 자주 빈번하게 가능하다.
*   **리소스/필드 제거**: **API 사용 중단 정책**을 따라야 한다.
*   **GA(일반 공급) API**: `v1` 버전에 도달하면 강력한 호환성 유지 약속을 한다.
*   **베타 API 버전**: 데이터 호환성을 유지하며, 기능이 안정화되면 GA API 버전을 통해 데이터를 변환하고 접근할 수 있도록 보장한다. 베타 API를 사용한다면 해당 API가 사용 중단 기간 동안 베타 또는 안정화 API 버전으로 전환해야 한다.
*   **알파 API 버전**: 호환성을 유지하려고 하지만 불가능한 경우도 있다. 알파 API를 사용한다면 업그레이드 시 릴리스 노트를 확인해야 한다.

### API 확장 방법
쿠버네티스 API는 두 가지 방식으로 확장할 수 있다.
1.  **커스텀 리소스(Custom Resources)**: API 서버가 선택한 리소스 API를 어떻게 제공해야 하는지를 선언적으로 정의한다.
2.  **애그리게이션 레이어(Aggregation Layer)**: 구현을 통해 확장한다.

### 정리
쿠버네티스 API는 클러스터 상태를 제어하는 **유일한 진입점**이다. **Discovery API**와 **OpenAPI**를 통해 API 사양을 공개하며, **API 그룹과 버전 관리**를 통해 호환성을 유지하면서 진화한다. GA API에 대한 강력한 호환성 약속이 있으며, **커스텀 리소스**나 **애그리게이션 레이어**를 통해 기능을 확장할 수 있다.
