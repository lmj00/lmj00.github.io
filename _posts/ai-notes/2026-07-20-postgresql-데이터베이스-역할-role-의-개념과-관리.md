> 한 줄 요약: PostgreSQL의 역할(Role)은 운영체제 사용자와 개념적으로 완전히 분리된 데이터베이스 클러스터 수준의 권한 주체이며, `CREATE ROLE` 명령어나 `createuser` 유틸리티로 생성하고 관리한다.

### 개요
PostgreSQL에서 **데이터베이스 역할(Role)**은 데이터베이스에 접근하고 작업을 수행할 수 있는 주체를 정의하는 개념이다. 이 역할은 운영체제의 사용자 계정과는 **개념적으로 완전히 별개**로 존재하며, 데이터베이스 클러스터 전체(개별 데이터베이스가 아닌)에 걸쳐 전역적으로 적용된다. 모든 데이터베이스 연결은 특정 역할의 이름으로 이루어지며, 이 역할은 해당 연결에서 실행되는 명령어의 초기 접근 권한을 결정한다.

### 역할의 개념과 운영체제 사용자와의 관계
공식 문서는 데이터베이스 역할과 운영체제 사용자를 명확히 구분한다. 두 개념은 서로 독립적이지만, 실제 운영에서는 대응 관계를 유지하는 것이 편리할 수 있다. 예를 들어, 많은 클라이언트 응용 프로그램(예: `psql`, `createuser`)은 기본적으로 현재 운영체제 사용자 이름을 연결 역할로 가정한다. 따라서 역할 이름과 운영체제 사용자 이름을 일치시키는 것이 관리상 편리할 수 있지만, **이는 필수 요구사항이 아니다**. 클라이언트 인증 설정(Chapter 20)에 따라, 클라이언트는 자신의 운영체제 사용자와 다른 역할로도 연결할 수 있다.

### 역할 생성과 관리
역할을 생성하는 기본적인 방법은 SQL 명령어 `CREATE ROLE`을 사용하는 것이다.
```sql
CREATE ROLE name;
```
여기서 `name`은 SQL 식별자 규칙을 따른다. 실제로는 `LOGIN`과 같은 추가 옵션을 명령어에 지정하는 경우가 많다. 역할을 제거할 때는 `DROP ROLE` 명령어를 사용한다.

명령줄에서 편의를 위해 제공되는 `createuser`와 `dropuser` 유틸리티는 위 SQL 명령어를 감싼 래퍼 프로그램이다.
```bash
createuser name
dropuser name
```

### 기존 역할 조회
현재 존재하는 역할 집합을 확인하려면 `pg_roles` 시스템 카탈로그를 조회하면 된다.
```sql
SELECT rolname FROM pg_roles;
```
로그인 가능한 역할만 보고 싶다면 `rolcanlogin` 컬럼으로 필터링할 수 있다. `psql` 내부 명령어 `\du`도 역할 목록을 보는 데 유용하다.

### 부트스트랩과 초기 역할
데이터베이스 시스템을 초기화(`initdb`)하면, 항상 하나의 미리 정의된 **로그인 가능한 역할**이 생성된다. 이 역할은 **슈퍼유저** 권한을 가지며, 별도로 지정하지 않으면 데이터베이스 클러스터를 초기화한 운영체제 사용자와 같은 이름을 가진다. 통상 이 역할의 이름은 `postgres`이다. 더 많은 역할을 생성하려면 먼저 이 초기 역할로 연결해야 한다.

### 역할의 중요성: 권한 결정
역할의 정체성은 연결된 클라이언트가 사용할 수 있는 권한 집합을 결정한다. 클라이언트가 연결 요청을 시작할 때 역할 이름을 지정하는 방식은 응용 프로그램에 따라 다르다(예: `psql`은 `-U` 옵션 사용). 따라서 다중 사용자 환경을 설정할 때는 권한을 신중하게 구성하는 것이 매우 중요하다.

### 정리
* PostgreSQL 역할은 운영체제 사용자와 독립된 데이터베이스 클러스터 수준의 엔티티이다.
* 역할은 `CREATE ROLE` SQL 명령어나 `createuser` 유틸리티로 생성하며, `pg_roles` 카탈로그나 `\du` 명령으로 조회할 수 있다.
* 모든 데이터베이스 연결은 특정 역할로 이루어지며, 이 역할이 해당 세션의 권한을 결정한다.
* 시스템 초기화 시 생성되는 슈퍼유저 역할(일반적으로 `postgres`)로 첫 연결 후 다른 역할을 생성할 수 있다.
* 역할 이름과 운영체제 사용자 이름을 일치시키는 것은 편의상일 뿐, 강제되는 것은 아니다.
