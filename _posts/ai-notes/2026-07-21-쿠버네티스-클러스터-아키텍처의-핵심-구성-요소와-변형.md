### 개요
쿠버네티스 클러스터는 컨테이너화된 애플리케이션을 운영하기 위한 플랫폼이다. 그 아키텍처는 **컨트롤 플레인**과 **워커 노드**라는 두 가지 핵심 요소로 나뉜다. 컨트롤 플레인은 클러스터에 대한 글로벌 결정을 내리고 상태를 관리하며, 워커 노드는 실제 애플리케이션 **파드**를 호스팅한다. 프로덕션 환경에서는 **내결함성**과 **고가용성**을 위해 여러 컴퓨터에 컨트롤 플레인을 분산시키고 여러 노드를 실행하는 것이 일반적이다.

### 클러스터의 기본 구조
쿠버네티스 클러스터는 최소 하나 이상의 워커 노드와 이를 관리하는 컨트롤 플레인으로 이루어진다. 워커 노드는 애플리케이션 워크로드의 구성 요소인 파드를 실행하는 머신이다. 컨트롤 플레인은 이러한 워커 노드와 파드를 관리하는 책임을 가진다.

### 컨트롤 플레인 구성 요소
컨트롤 플레인은 클러스터의 두뇌 역할을 하는 여러 컴포넌트로 구성된다. 이들은 스케줄링, 상태 감지, 이벤트 응답 등의 글로벌 결정을 담당한다.

**kube-apiserver**
쿠버네티스 API를 노출하는 **컨트롤 플레인의 프론트엔드**다. 사용자, CLI, 다른 구성 요소의 모든 요청은 API 서버를 통해 처리된다. 수평 확장이 가능하도록 설계되어 여러 인스턴스를 배포하고 트래픽을 분산시킬 수 있다.

**etcd**
모든 클러스터 데이터의 **일관되고 고가용성의 키-값 저장소**로 사용된다. 쿠버네티스 클러스터의 상태 정보가 여기에 저장되므로, 데이터 백업 계획이 필수적이다.

**kube-scheduler**
새로 생성되었지만 노드가 할당되지 않은 **파드를 감시**하고, 실행할 적절한 노드를 선택하는 컴포넌트다. 스케줄링 결정 시 리소스 요구사항, 하드웨어/소프트웨어 제약, 어피니티/안티-어피니티 명세, 데이터 지역성, 워크로드 간 간섭, 데드라인 등을 고려한다.

**kube-controller-manager**
**컨트롤러 프로세스**를 실행하는 컴포넌트다. 논리적으로 각 컨트롤러는 별도의 프로세스지만, 복잡성을 줄이기 위해 단일 바이너리로 컴파일되어 단일 프로세스에서 실행된다. 주요 컨트롤러에는 노드 컨트롤러(노드 다운 감지 및 응답), 잡 컨트롤러(일회성 작업 관리), EndpointSlice 컨트롤러(서비스와 파드 연결), ServiceAccount 컨트롤러(기본 서비스 계정 생성) 등이 있다.

**cloud-controller-manager**
클라우드 공급자별 제어 로직을 포함하는 컴포넌트다. 클러스터를 클라우드 공급자의 API에 연결하며, 클라우드 플랫폼과 상호작용하는 부분을 클러스터 내부 로직과 분리한다. 온프레미스나 학습 환경에서는 일반적으로 실행되지 않는다. kube-controller-manager와 마찬가지로 여러 논리적 제어 루프를 단일 프로세스로 결합하며, 성능이나 내결함성을 위해 수평 확장할 수 있다. 노드 컨트롤러(클라우드에서 노드 삭제 확인), 라우트 컨트롤러(클라우드 인프라 라우트 설정), 서비스 컨트롤러(클라우드 로드 밸런서 관리) 등이 클라우드 의존성을 가질 수 있다.

### 노드 구성 요소
노드 구성 요소는 모든 워커 노드에서 실행되며, 파드를 유지하고 쿠버네티스 런타임 환경을 제공한다.

**kubelet**
각 노드에서 실행되는 **에이전트**로, 파드 내의 컨테이너가 실행 중이고 건강한지 확인한다. 다양한 메커니즘을 통해 제공된 PodSpec 집합을 받아, 그에 기술된 컨테이너가 실행되도록 보장한다. 쿠버네티스가 생성하지 않은 컨테이너는 관리하지 않는다.

**kube-proxy (선택 사항)**
각 노드에서 실행되는 **네트워크 프록시**로, 쿠버네티스 **서비스** 개념의 일부를 구현한다. 노드의 네트워크 규칙을 유지 관리하여 클러스터 내부 또는 외부의 네트워크 세션에서 파드로의 통신을 가능하게 한다. 운영 체제의 패킷 필터링 계층을 사용할 수 있으면 이를 사용하고, 그렇지 않으면 직접 트래픽을 포워딩한다. 일부 네트워크 플러그인은 서비스를 위한 패킷 포워딩을 자체 구현하여 kube-proxy와 동등한 동작을 제공할 수 있는데, 이 경우 노드에서 kube-proxy를 실행할 필요가 없다.

**컨테이너 런타임**
쿠버네티스가 컨테이너를 효과적으로 실행할 수 있게 하는 **기본 구성 요소**다. 쿠버네티스 환경 내에서 컨테이너의 실행과 생명주기를 관리한다. containerd, CRI-O 및 **Kubernetes CRI(Container Runtime Interface)** 를 구현한 다른 런타임을 지원한다.

### 애드온
애드온은 DaemonSet, Deployment 등의 쿠버네티스 리소스를 사용하여 클러스터 수준의 기능을 구현한다. 클러스터 수준 기능을 제공하므로, 애드온의 네임스페이스 리소스는 일반적으로 `kube-system` 네임스페이스 내에 속한다.

| 애드온 | 역할 |
| :--- | :--- |
| **DNS** | 거의 모든 쿠버네티스 클러스터에 필요한 핵심 애드온. 쿠버네티스 서비스에 대한 DNS 레코드를 제공하는 DNS 서버다. |
| **웹 UI (대시보드)** | 웹 기반 UI로, 클러스터 및 애플리케이션 관리와 문제 해결을 제공한다. |
| **컨테이너 리소스 모니터링** | 컨테이너에 대한 일반 시계열 메트릭을 중앙 DB에 기록하고 탐색 UI를 제공한다. |
| **클러스터 수준 로깅** | 컨테이너 로그를 중앙 로그 저장소에 저장하고 검색/탐색 인터페이스를 제공한다. |
| **네트워크 플러그인** | CNI(Container Network Interface) 명세를 구현하여 파드에 IP 주소를 할당하고 클러스터 내 통신을 가능하게 한다. |

### 아키텍처 변형과 유연성
쿠버네티스의 핵심 구성 요소는 일관되지만, 배포 및 관리 방식은 요구사항에 따라 다양할 수 있다.

**컨트롤 플레인 배포 옵션**
*   **전통적 배포**: 전용 머신이나 VM에서 systemd 서비스로 직접 실행.
*   **정적 파드**: 특정 노드의 kubelet이 관리하는 정적 파드로 배포 (kubeadm에서 일반적).
*   **셀프 호스팅**: 컨트롤 플레인이 쿠버네티스 클러스터 자체 내의 파드로 실행됨.
*   **관리형 쿠버네티스 서비스**: 클라우드 공급자가 컨트롤 플레인을 추상화하여 관리.

**워크로드 배치 고려사항**
*   작은 개발 클러스터에서는 컨트롤 플레인과 사용자 워크로드가 동일 노드에서 실행될 수 있다.
*   대규모 프로덕션 클러스터는 전용 노드를 컨트롤 플레인에 할당하여 사용자 워크로드와 분리하는 경우가 많다.
*   일부 조직은 중요한 애드온이나 모니터링 도구를 컨트롤 플레인 노드에서 실행하기도 한다.

**확장성과 커스터마이제이션**
쿠버네티스 아키텍처는 상당한 수준의 커스터마이제이션을 허용한다.
*   기본 스케줄러와 함께 또는 대체하여 **커스텀 스케줄러**를 배포할 수 있다.
*   API 서버는 **CustomResourceDefinitions(CRD)** 와 **API Aggregation**으로 확장할 수 있다.
*   클라우드 공급자는 **cloud-controller-manager**를 사용해 쿠버네티스와 깊게 통합할 수 있다.
이러한 유연성을 통해 조직은 운영 복잡성, 성능, 관리 오버헤드 등의 요소를 균형 있게 고려하여 클러스터를 특정 요구사항에 맞게 조정할 수 있다.

### 정리
쿠버네티스 클러스터는 **관리(컨트롤 플레인)** 와 **실행(노드)** 의 명확한 분리를 기반으로 한다. 컨트롤 플레인의 핵심 구성 요소(kube-apiserver, etcd, 스케줄러, 컨트롤러 매니저)는 클러스터의 상태와 논리를 관리하며, 노드의 구성 요소(kubelet, 컨테이너 런타임, kube-proxy)는 애플리케이션 워크로드를 안정적으로 실행한다. DNS 같은 필수 애드온과 다양한 네트워크 플러그인을 통해 완전한 기능을 갖춘다. 또한 전통적 배포부터 관리형 서비스에 이르는 다양한 배포 모델과 커스터마이제이션 옵션을 제공함으로써, 다양한 규모와 요구사항에 맞춰 유연하게 적용될 수 있는 아키텍처를 지닌다.
